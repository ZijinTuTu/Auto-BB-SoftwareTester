# 四边形覆盖问题黑盒测试实验 README

## 1. 实验目标

本实验围绕“**四边形覆盖问题**”开展黑盒测试。题目给定两个四边形（在项目中实际建模为与坐标轴平行的矩形），输入形式为：

- 四边形 1：`(X1Coord, Y1Coord, Width1, Height1)`
- 四边形 2：`(X2Coord, Y2Coord, Width2, Height2)`

其中：

- `XCoord`、`YCoord` 表示左上角坐标
- `Width`、`Height` 表示宽和高

程序输出为：**两个四边形的覆盖关系**。

本实验的核心要求包括：

1. 使用**等价类划分**和**边界值分析**设计测试用例。
2. 使用 `box-black-box` 项目中的可视化界面对这些测试用例进行黑盒测试。
3. 分析项目中给出的多个判定函数是否存在 Bug。
4. 使用生成式 AI 自动生成测试用例，并与人工设计用例进行对比分析。

## 2. 项目文件说明

在本实验目录中，主要涉及以下几类文件。

### 2.1 原始项目文件

#### `index.html`
网页入口文件，负责构建测试界面。

它的作用主要包括：

- 展示测试用例列表
- 提供输入框，让用户输入两个四边形参数
- 提供 `Run this test case` 和 `Run all test cases` 按钮
- 展示每个算法的输出结果

需要注意的是：`index.html` 只是**前端界面外壳**，并不直接决定矩形覆盖关系的判定逻辑。真正的业务逻辑在 JavaScript 文件中。

#### `js/app.js`
AngularJS 应用入口，用于创建前端模块。

#### `js/controllers/BoxTestController.js`
控制器文件，负责：

- 管理测试用例数据
- 响应页面中的“运行测试”操作
- 调用各个算法函数进行判定
- 将判定结果渲染回页面

也就是说，页面中点击“运行测试”后，最终是控制器去调用算法并更新界面。

#### `js/services/algorithms.js`
本实验最关键的源码文件。

这个文件中定义了多个用于判断“四边形是否覆盖”的函数（页面中显示为多种不同算法）。

这些函数的目标都是：

> 当且仅当两个四边形存在**非零面积重叠**时，返回 `true`；否则返回 `false`。

本实验黑盒测试的主要对象，就是这个文件中的多个算法实现。

---

### 2.2 自动测试与实验辅助文件

#### `rect_blackbox_runner.py`
这是本实验的**核心自动测试脚本**。

它的作用是：

- 读取或内置人工设计的测试用例
- 直接调用各个矩形覆盖判定算法
- 将算法输出与预期结果进行比对
- 输出每个测试用例下各算法的通过/失败情况
- 汇总每个算法被哪些测试用例检出 Bug

这个脚本的优势是：

- 不依赖浏览器页面是否正常渲染
- 不依赖 AngularJS 或 CDN 资源加载成功
- 能更稳定地验证算法逻辑本身

#### `test_cases.json`
人工设计的测试用例数据文件。

建议将测试用例按如下信息组织：

- 用例编号
- 用例名称
- 两个四边形参数
- 预期结果
- 设计方法（等价类 / 边界值）
- 设计目的（希望验证什么，或针对哪个潜在缺陷）

#### `qwen_generate_cases.py`
用于调用通义千问 API，自动生成测试用例。

它的作用是：

- 从环境变量中读取 API Key
- 向大模型发送提示词
- 让模型按要求输出测试用例
- 将 AI 生成的结果保存成 JSON 文件，方便与人工设计用例比较

#### `.env`
环境变量配置文件，用于存放 API Key。

例如：

```env
DASHSCOPE_API_KEY=真实APIKey
```

这样可以避免把 API Key 直接写进代码中，提高安全性。

#### `prompt.txt`
保存用于调用通义千问的提示词。

实验报告中通常需要写明“使用了什么提示词”，所以把提示词单独保存为文本文件，是比较规范的做法。

## 3. 实验中的测试思想设计

本实验要求使用两种经典黑盒测试方法：

- 等价类划分
- 边界值分析

### 3.1 等价类划分

等价类划分的思路是：

把输入数据按“对程序行为有相同影响”的原则划分为若干类，每一类中选取少量代表值进行测试。

对于本题，可以将输入划分为：

#### 3.1.1 有效等价类

1. **普通重叠**
   - 两个四边形存在非零面积交集
   - 预期输出：`True`

2. **完全分离**
   - 两个四边形在水平或垂直方向上互不相交
   - 预期输出：`False`

3. **一个四边形完全包含另一个**
   - 存在覆盖关系，且交叠面积非零
   - 预期输出：`True`

4. **两个四边形完全相同**
   - 完全重合
   - 预期输出：`True`

5. **十字形重叠**
   - 两个四边形交叉形成“十字形”区域
   - 预期输出：`True`

#### 3.1.2 无效等价类

1. `Width <= 0`
2. `Height <= 0`
3. 坐标为负数
4. 非整数输入

在题目描述中，四边形参数要求是整数，且宽高必须大于 0，因此这些输入都属于无效输入。

---

### 3.2 边界值分析

边界值分析强调：

> 程序最容易在“刚好达到临界条件”的位置出错。

对于四边形覆盖问题，关键边界包括：

1. **刚好边接触**
   - 例如：`x1 + width1 = x2`
   - 两个四边形只有公共边，没有面积重叠
   - 预期输出：`False`

2. **刚好角接触**
   - 例如一个四边形右下角恰好接触另一个四边形左上角
   - 只有公共点，没有面积重叠
   - 预期输出：`False`

3. **完全重合**
   - 属于极端覆盖情况
   - 预期输出：`True`

4. **刚好穿插形成十字形重叠**
   - 有非零面积重叠，但不一定有顶点落入对方内部
   - 很容易暴露“只检查顶点是否在内部”的错误算法

因此，本实验的边界值测试重点要围绕“面积是否大于 0”这个核心边界来设计。

## 4. 人工测试用例设计原则

### 4.1 设计目标

人工设计测试用例时，不仅要覆盖题目要求的输入情况，还要尽量覆盖**容易出错的特殊关系**。

因此测试集应至少包括：

- 普通重叠
- 完全分离
- 水平分离
- 垂直分离
- 一个包含另一个
- 两者完全相同
- 边接触
- 角接触
- 十字形重叠
- 非法输入

### 4.2 为什么这些用例重要

这些用例能有效区分常见错误算法，例如：

- 有的算法把“边接触”误判为覆盖
- 有的算法只能识别“顶点落入内部”的重叠，无法识别十字形交叉
- 有的算法对完全重合的情况处理错误
- 有的算法只考虑一种矩形顺序，交换两个输入后结果会变化

所以这些用例既符合黑盒测试思想，也能有效用于 Bug 检出。

## 5. 自动测试代码设计原理

### 5.1 为什么不只依赖网页界面测试

最初我考虑用 Selenium 自动打开 `index.html`，自动填写输入框并读取页面结果。

但在本实验里，这种方法存在两个问题：

1. 页面依赖 AngularJS、Angular Material 等外部资源
2. 一旦外部脚本加载不完整，页面会停留在模板状态，例如出现 `{{test.errors}}`

这样会导致：

- 页面能打开
- 输入框能填写
- 按钮能点击
- 但结果不一定可靠

因此，为了更稳定地完成实验，本实验采用了更可靠的方法：

> **直接对算法函数进行自动测试，而不是完全依赖前端页面。**

---

### 5.2 `rect_blackbox_runner.py` 的设计思路

这个脚本的核心思路是：

1. 定义统一的数据结构表示四边形
2. 将项目中的多个判定函数封装为 Python 可调用对象
3. 读取人工设计测试用例
4. 对每个测试用例，依次调用各算法
5. 将实际输出与期望输出进行比较
6. 生成测试结果汇总

这样做的优点是：

- 运行稳定
- 不受浏览器环境影响
- 结果可重复
- 便于统计每个算法的缺陷分布

### 5.3 自动测试脚本通常包含的模块

#### （1）测试数据模块
保存测试用例，例如：

- `box1`
- `box2`
- `expected`
- `case_id`
- `method`
- `purpose`

#### （2）算法调用模块
把待测算法组织成字典，例如：

```python
ALGORITHMS = {
    "a": alg_a,
    "b": alg_b,
    "c": alg_c,
    "d": alg_d,
    "e": alg_e,
    "f": alg_f,
}
```

#### （3）执行模块
循环执行：

- 外层遍历测试用例
- 内层遍历算法

#### （4）比对模块
将算法输出与 `expected` 进行比较：

- 相同：通过
- 不同：失败

#### （5）汇总模块
统计：

- 每个算法一共失败了几次
- 是被哪些测试用例检出的
- 哪些测试用例最有效

## 6. 本实验中自动测试文件的推荐组织方式

推荐在实验目录中采用如下结构：

```text
box-black-box-gh-pages/
│
├─ index.html
├─ README.md

├─ rect_blackbox_runner.py
```

## 7. AI 测试用例生成模块设计原理

### 7.1 设计目标

使用通义千问 API，让大模型根据提示词自动生成测试用例。

要求模型生成的内容应尽量满足：

- 使用等价类划分和边界值分析
- 输出结构统一
- 包含预期结果
- 包含测试目的说明

### 7.2 为什么要把 API Key 放进 `.env`

这样做有三个好处：

1. 避免把密钥硬编码到源码中
2. 便于不同机器迁移
3. 便于在报告中展示代码时隐藏敏感信息

### 7.3 调用流程

`qwen_generate_cases.py` 的典型流程：

1. 使用 `python-dotenv` 加载 `.env`
2. 读取 `DASHSCOPE_API_KEY`
3. 读取 `prompt.txt` 中的提示词
4. 调用通义千问接口
5. 让模型返回 JSON 形式的测试用例
6. 将结果保存到文件中

### 7.4 AI 生成用例与人工设计用例的差异

一般来说：

- **人工设计用例**更有针对性，特别擅长覆盖特定边界和缺陷模式
- **AI 生成用例**覆盖面可能更广，但容易出现重复、泛化过度、边界不够精准的问题

因此更合理的实验结论是：

> 人工设计用例适合构建核心测试集，AI 生成用例适合辅助扩展与补充。

## 9. 运行说明

### 9.1 运行人工设计用例自动测试

```bash
python rect_blackbox_runner.py
```

### 9.2 运行 AI 测试用例生成脚本

先安装依赖：

```bash
pip install openai python-dotenv
```

配置 `.env`：

```env
DASHSCOPE_API_KEY=APIKey
```

再执行：

```bash
python qwen_generate_cases.py
```

## 10. 总结

本实验的关键：

1. 是否基于黑盒测试思想设计了高质量测试用例
2. 是否能用这些用例有效检出算法缺陷
3. 是否能对人工设计与 AI 生成测试用例进行合理对比分析

因此，本实验代码设计可以概括为三层：

- **界面层**：`index.html` + Angular 页面，用于可视化展示
- **逻辑层**：`algorithms.js`，实现多个矩形覆盖判定算法
- **实验层**：`rect_blackbox_runner.py`、`test_cases.json`、`qwen_generate_cases.py`，用于完成黑盒测试与 AI 辅助测试

这三层共同构成了本实验的完整实现。
